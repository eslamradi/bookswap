## Work Items

1. **`profiles` table migration** (small) — Supabase Auth owns `auth.users`; app-specific fields (display name, address-masking default from Scenario 05) live in a `public.profiles` row keyed by `auth.users.id`.
2. **Supabase client helpers** (small) — ✅ done: `src/lib/supabase/client.ts` (browser), `src/lib/supabase/server.ts` (server components/actions)
3. **`/signup` route — page shell** (small) — logo, headline, centered narrow container. Matches prototype 1:1.
4. **Auth actions component** (medium) — Google button, Apple button, email input + continue (magic link), loading state, inline error state. Client component (needs interactivity).
5. **`/auth/callback` route** (medium) — required by Supabase Auth to exchange the OAuth/magic-link code for a session, then redirect onward.
6. **Post-auth redirect** (small) — redirect to `/listings/new` (Batch Listing — not built yet in this pass, will 404 until it exists, same as the prototype's honest gap).

## Dependencies / Order

1 → 2 (done) → 3 → 4 → 5 → 6. Table before UI; shell before interactive auth actions; callback route before anything can redirect through it.

## External Setup Required (outside this codebase)

- A real Supabase project must exist, with `NEXT_PUBLIC_SUPABASE_URL` / `NEXT_PUBLIC_SUPABASE_ANON_KEY` filled into `.env.local` (copy from `.env.local.example`).
- Google and Apple OAuth providers must be configured in the Supabase dashboard (Authentication → Providers) for those two buttons to actually work — this cannot be done from code. Email magic link works with Supabase's built-in email out of the box in dev.
- Migration in `supabase/migrations/` must be applied to the project (via Supabase CLI or dashboard SQL editor).

## Decisions Made During Implementation

- **Route:** mounted at `/` (not `/signup`) — every entry point lands here directly.
- **Email auth:** magic link, not OTP code — matches the spec's one-field wireframe. Added a "Check your email" success state (not in the original spec's States table, since the tab that sent the link can't know when it's clicked) — **follow-up: add this state to the page spec.**
- **Framework note:** this Next.js version deprecated `middleware.ts` in favor of `proxy.ts` (renamed file + exported function) — used `proxy.ts` throughout.

## Bugs Found and Fixed During Verification

1. `createServerClient()`/`createBrowserClient()` throw **synchronously** when Supabase env vars are missing/empty — not just on the later network call. `src/lib/supabase/middleware.ts` was 500-ing every request before an explicit env-var guard was added. `src/components/auth/sign-in-actions.tsx` had no try/catch around `createClient()`, so a click would leave the button stuck in loading state forever with no visible error. Both fixed; confirmed via `curl` that `/` (200), `/auth/callback` with no code, with an OAuth error param, and with a fake code all behave correctly (redirect to the right place, no 500s) with zero Supabase configuration present.

## Acceptance Criteria (from spec)

- [x] Passwordless — no password field anywhere
- [x] No profile setup / optional fields beyond auth itself
- [x] Social login (Google, Apple) — one tap each
- [x] Email fallback — single field + continue (magic link, not password)
- [x] Loading state shown only on the clicked method's button
- [x] Error state shown inline, method stays selected (email value preserved), retry available
- [x] No marketing/upsell content beyond the one headline
- [x] Narrow, centered, responsive layout (mobile → desktop) — Tailwind classes ported from the validated prototype

**Verified via:** `npm run lint`, `npm run build` (both clean), `npm run dev` + curl checks against `/` and `/auth/callback` edge cases with zero Supabase configuration. **Not yet verified:** actual OAuth/magic-link round trip — requires a real Supabase project (see External Setup Required above).
