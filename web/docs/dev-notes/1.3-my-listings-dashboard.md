# Dev Plan: 1.3 My Listings/Dashboard (`/listings`)

**Spec:** [design-artifacts/C-UX-Scenarios/01-toms-five-minute-declutter/1.3-my-listings-dashboard/1.3-my-listings-dashboard.md](../../design-artifacts/C-UX-Scenarios/01-toms-five-minute-declutter/1.3-my-listings-dashboard/1.3-my-listings-dashboard.md)
**Branch:** `feature/1.3-my-listings-dashboard`

## Decisions

- **Real dashboard, not just a post-submit confirmation screen.** The spec frames this as "confirm N books were just listed," but a real product needs this route to work correctly on every visit (bookmark, refresh, return later) — not only immediately after a batch submit. So: query all of the signed-in user's listings on every load, not just "whatever was in a just-submitted count." The confirmation-banner framing still applies naturally (it reads "✓ N books listed" using the real count), it's just sourced from a real query instead of session/redirect state.
- **Server component, no client interactivity needed** — this page only reads and displays; "Add more books" is a plain link to `/listings/new`. Simpler than 1.1/1.2, no `"use client"` boundary required.
- **Photo display:** `listing-photos` bucket is private (owner-scoped RLS), so a signed URL is generated server-side per listing that has a `photo_path`; falls back to the Open Library `cover_url` if no photo was attached.

## Files

- `src/app/listings/page.tsx` — auth-guarded server component: query listings, resolve signed URLs, render.

## Acceptance Criteria (from spec, adapted per the decision above)

- [x] Auth-guarded — redirects unauthenticated visitors to `/`
- [x] Shows real count and list of the signed-in user's listings, freshly queried (not stale/cached from submit)
- [x] Each row shows thumbnail (signed URL for uploaded photos, Open Library cover as fallback, generic placeholder if neither), title, author, status
- [x] Empty state when the user has no listings yet
- [x] "Add more books" links to `/listings/new`
