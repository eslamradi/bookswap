## Decisions Made (flagged, not silently assumed)

1. **ISBN lookup:** [Open Library's Books API](https://openlibrary.org/dev/docs/api/books) — free, keyless, no rate-limit setup needed. The prototype simulated this against an 8-book demo catalog; production needs a real source.
2. **Manual ISBN entry, not live camera barcode scanning.** The prototype already simulated scanning as manual entry (no camera available in a static prototype). For real production, actual camera-based barcode auto-detection is a legitimate future enhancement (`BarcodeDetector` API or a library like `@zxing/browser`), but it's a meaningfully larger scope item than this page needs to ship with — manual ISBN entry is a fully real, working feature on its own, not a fake stand-in. Flagging this as a deliberate scope line, not an oversight.
3. **Condition photo — real file upload**, not the prototype's toggle simulation. `<input type="file" accept="image/*" capture="environment">` (works for both camera capture and gallery pick on mobile), uploaded to Supabase Storage on final submit.
4. **Queue persistence:** metadata (isbn/title/author/condition) mirrored to `localStorage` on every change, same mechanism the validated prototype used, so refreshing mid-batch doesn't lose Tom's progress (his single highest-scored fear, 15/15, is exactly this). **Disclosed gap:** attached photo *files* can't be serialized into `localStorage` — if the tab is refreshed mid-batch, book metadata survives but photos would need to be re-attached. Acceptable for this pass; a more robust fix (draft rows in the DB, or IndexedDB for file blobs) is a future improvement if this turns out to matter in practice.
5. **Schema scope:** `listings` table gets owner-only RLS (select/insert/update on own rows) — no public-read policy yet, matching the same "don't build for features that don't exist yet" discipline as the `profiles` table in 1.1. Browse Listings (Scenario 03) doesn't exist yet; add public-read when it's actually built.
6. **Route:** `/listings/new` (already the `DEFAULT_REDIRECT_PATH` from 1.1). Dashboard (1.3, next page) will live at `/listings`.

## Schema

New migration: `listings` table (id, owner_id, isbn, title, author, cover_url, condition, status, created_at) + RLS (owner-scoped) + `listing-photos` Storage bucket (owner-scoped read/write for now).

## Files

1. `supabase/migrations/<ts>_create_listings.sql` — schema + storage bucket/policies
2. `src/lib/books/lookup.ts` — Open Library ISBN lookup (`fetch` wrapper, typed result or `null`)
3. `src/app/listings/new/page.tsx` — server component, auth guard (redirect to `/` if no session)
4. `src/components/listings/batch-listing-form.tsx` (`"use client"`) — the queue UI: ISBN input + lookup, per-item condition select + photo input + remove, live count header, bulk submit with loading/error states, localStorage-backed queue persistence

## Acceptance Criteria (from spec)

- [x] Auth-guarded — redirects unauthenticated visitors to `/`
- [x] ISBN entry auto-fills title/author via real lookup; graceful "not found" state for unrecognized ISBNs
- [x] Per-book condition preset (Good/Fair/Worn), no free text
- [x] Per-book photo attach (real file input)
- [x] Queue survives a page refresh (metadata, not photos — disclosed above)
- [x] Bulk submit creates real `listings` rows + uploads photos, single action for the whole queue
- [x] Loading state on submit; error state preserves the queue (nothing lost on failure)
- [x] Redirects to `/listings` (Dashboard) on success — confirmed 404 there until 1.3 is built, as expected
